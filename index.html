Introduction

Vous decouvrirez à travers la lecture de cette page l'outil de travail gratuit et open sourced nommé Git ainsi que toutes ses particularités qui en fait un outil essentiel pour tout travail de programmation.

Mais qu'est-ce que Git ?

Git est un outil de gestion de code source permettant un historique de version d'un programme ou tout autres fichiers de texte brut. Git organise les version en faisant des "noeuds" (appelés commits) contenant les informations des modifications depuis le dernier commit créé. Ceci permet de plus de savoir plus facilement les modifications apportées à chacune des "versions" puisque chaque commit doit être définit avec un résumé des modifications apportées. Finalement, chaque commit contient de plus les informations de l'auteur de ce dernier, ainsi, on peut savoir facilement qui a fait quoi lorsqu'on travail en équipe.

Vous pouvez télécharger / installer git ici!

Un bref aperçu avant de plonger dans le coeur de git

Commençant par la gauche, chacun des cercles ici est un commit. Les commits qui divergent de la ligne du millieu forments des "branches", un concept permettant de travailler en parallèle sur différentes tâches sans affecter la version principale.

Ne vous inquiétez pas, vous allez comprendres pourquoi les commits font de tels chemins en continuant votre lecture!

Un système distribué

Git est un système distribué, ce qui signifie qu'il peut communiquer avec d'autres instances pour synchroniser les changements effectués par différents collègues. Ceci est dû au fait que toutes les informations de git sont dans un dossier local du projet : quand le dossier du projet est gérer par git, on appele ce dossier un repository. De plus, grâce à ce système, ce repository peut être transféré sur un serveur distant et ainsi, chaque personne peut travailler à distance et tout de même avoir un accès au modifications effectuées par les autres : ils n'ont qu'à faire un lien entre leur repository locale et le repository mis sur le server.

Oui, cela veut aussi dire que l'on peut travailler sur le projet sans avoir de connection à l'internet, puisque tout les fichiers sont sur notre machine!
Évidemment, pour ensuite synchroniser les changements avec ceux des collègues, il faudra une connection quelconque avec leur travail, mais cela n'est pas obligé d'être à travers l'internet : dès qu'il y a une connection entre les deux machines, la synchronisation sera possible (donc LAN, serveur interne, etc sont aussi des solutions possibles).

Les branches

Le concepte des branches est très important dans tout projet git. En fait, dès que le projet est gérer par git, ce projet est dans une branche. Comme vous avez vu dans l'image plus haut, un projet git ressemble beaucoup à des branches d'un arbre, certaines sortant du tronc de l'arbre et d'autres des branches elles-mêmes. En gardant l'image de l'arbre en tête, la logique des branches dans git devient très facile à comprendre : une branche sert à faire avancer un projet sans apporter d'impacts à la branche principale, donc de pouvoir travailler en parallèle.

Travailler en parallèle, ça sert à quoi?

Ceci permet au projet d'avoir des versions finales qui sont différentes les unes des autres. En effet, on peut penser à un programme qui a une version stable, une version beta et même une version alpha : toutes ces versions pourraient être sur des branches différentes, faisant en sorte que la branche stable n'est pas affectée par le dévelopement effectué dans la branche alpha, dans cet exemple. Cette séparation de version permet ainsi un plus grand contrôle sur les mises à jour des programmes, tout en laissant place au travail effectué sur ce même programme en parallèle.

La liberté dans la gestion des branches

Grâce à l'implémentation abstraite du concepte des branches dans le système de git, chaque équipe peut décider comment elles gèrent leur branches, ce qui leur permettra d'avoir leur propre flux de travail. Plusieurs modèles des branchage existe déjà, étant conçues pour être plus productif, réduire les problèmes de synchronisation ou simplement aider les équipes à avoir une méthode de travail cohérente et unifiée. Plus de détails sur les modèles de branches sont discutés dans cette page.

Un aperçu visuel du fonctionnement de git

Avec les commandes ci-haut, voici à quoi ressemble le fonctionnement de git, pour les visuels!

Il est ici plus facile de comprendre le fonctionnement de git en ayant en tête les commandes :

  1. On fait des modifications dans le workspace (le dossier local du projet);
  2. On ajoute (add) ces modifications dans l'index, une place qui prépare les modifications à être placées dans un commit;
  3. On effectue un commit, préférablement avec un message utile, qui lui va aller dans le repository locale;
  4. On effectue un push vers le repository partagé;
  5. (optionnel) On peut fetch pour dire à notre repository locale s'il y a des modifications dans le repository partagé;
  6. Dans le cas où des modifications sont présentes dans le repository partagé, on peut pull pour ajouter les modifications distantes dans notre workspace et donc avoir les nouvelles modifications localement.

Les gestionnaires visuels pour git

Pour ceux qui ne sont pas habitués avec la ligne de commandes, il existe des programmes offrant une gestion des projets git sous forme d'interfaces où on doit cliquer pour éxécuter des commandes. Ceci permet premièrement d'aider à apprendre git, puisqu'il est plus facile de se souvenir de ce qu'un bouton fait plutôt que de se souvenir toutes les commandes que git offre (il y en as beaucoup!). Deuxièmement, il est pratique d'utiliser de tel programmes pour voir plus rapidement et facilement l'arbre d'un projet et d'accéder aux informations ainsi plus rapidement : moins de temps de gestion, plus de temps de programmation, plus de temps pour avancer le projet.

Évidemment, plusieurs programmes existes, tel que GitKraken, SourceTree ou même GitHub Desktop. Plus de détails se trouve sur cette page.

Services de gestion de repository externes

Il existe plusieurs services qui permettent de contenir vos repositories pour faciliter le partage entre équipes et ajouter des fonctionnalités à vos projets. Entre autres, vous y trouverez GitHub, BitBucket, GitLab, Team Foundation Servers et bien d'autres moins populaires chacuns ayant leurs propres caractéristiques.

Ces services peuvent entre autres :

  • Gérer la documentation du projet;
  • Gérer la gestion des problèmes du projet ainsi que les demandes clients;
  • Intégrer des services tierces pour, par exemple, rouler des tests automatiquement à chaque commit ou vérifier la structure des fichiers;
  • Générer des rapports basés sur la fréquence des commits ou le nombre de problèmes ouverts / fermés, par exemple;
  • et bien plus encore!

Vous trouverez plus d'information sur ces services sur cette page.

Conclusion

En conclusion, git nous permet de gérer un projet en nous donnant un contrôle sur les versions, une abilitée de synchroniser les modifications de manière très facile ainsi que d'intégrer notre projet avec des services externes nous offrant encore plus de flexibilité pour produire un produit complet et fonctionnel en équipe.

En fait, nous ne sommes pas obligé d'être en équipe pour utiliser git : même étant seul, nos projets sont plus faciles à gérer grâce à git et cela nous permet d'être plus productifs.

Ce document même est en fait géré par git et mis sur GitHub à cette page où l'on peut voir le progrès que j'ai effectué sur ce rapport, ainsi que l'intégration d'un validateur HTML externe (qui m'a d'ailleurs aidé plusieurs fois!) - allez y donner un coup d'oeil!
